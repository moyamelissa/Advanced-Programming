# Rapport – Travail 2 : Maîtrise de Git pour la collaboration

## Contexte
Ce travail consistait à simuler un workflow Git collaboratif autour d'une application Go (`word-stats`) capable de générer des statistiques textuelles à partir d'une chaîne de caractères.

## Workflow Git
Le projet a débuté par l'initialisation d'un dépôt local avec `git init`, suivie du renommage de la branche principale en `main` via `git branch -M main`. Le fichier `main.go` initial ne contenait qu'une seule fonction, `countLines`, qui s'appuie sur `strings.Split` pour compter les lignes séparées par le caractère `\n`. Ce premier état fonctionnel a été validé avec `git add` et `git commit`. À partir de `main`, deux branches de fonctionnalités ont été créées avec `git checkout -b`. La première, `count-words`, a introduit la fonction `countWords` utilisant `strings.Fields` pour découper le texte selon tout type d'espace blanc. La seconde, `count-chars`, a ajouté la fonction `countChars`, qui parcourt chaque rune du texte et exclut les espaces, tabulations et sauts de ligne (`' '`, `'\n'`, `'\r'`, `'\t'`). Chaque branche a fait l'objet d'au moins un commit dédié. La branche `count-words` a ensuite été fusionnée dans `main` via `git merge count-words`. Puisque `main` n'avait pas divergé, cette fusion s'est effectuée en _fast-forward_, sans conflit.

## Résolution du conflit de fusion
Afin de provoquer un conflit intentionnel, la fonction `main()` a été modifiée de manière divergente dans deux branches distinctes. Sur `main`, après le merge précédent, `main()` n'affichait que le comptage des mots via `fmt.Printf`. Sur `count-chars`, elle n'affichait que le comptage des caractères. Ces deux modifications portaient sur les mêmes lignes, rendant la fusion automatique impossible. Lors de `git merge count-chars`, Git a donc inséré les marqueurs `<<<<<<<`, `=======` et `>>>>>>>` dans `main.go`. La résolution a été effectuée manuellement : ouverture du fichier dans un éditeur, suppression de tous les marqueurs, puis combinaison des deux appels `fmt.Printf` dans `main()` afin d'afficher simultanément le nombre de mots et le nombre de caractères. La résolution a été finalisée avec `git add main.go` suivi de `git commit -m "Résolution du conflit: affichage mots + caractères"`.

## Défis rencontrés
Le premier défi a été de déterminer l'ordre optimal des fusions. Fusionner `count-words` en premier garantissait un _fast-forward_ propre, tandis que la fusion subséquente de `count-chars` produisait le conflit attendu par l'énoncé ; inverser cet ordre aurait modifié la nature du conflit et compliqué la démonstration du scénario demandé. Par ailleurs, la résolution manuelle des marqueurs a exigé une lecture attentive du fichier pour identifier précisément les blocs en conflit, comprendre quelle version provenait de quelle branche, et conserver les deux fonctionnalités sans introduire de duplication de code ni de régression fonctionnelle. L'avertissement `LF will be replaced by CRLF`, fréquent sous Windows lors des `git add`, a initialement suscité une inquiétude, mais s'est avéré bénin et sans incidence sur l'exécution du programme Go ni sur les résultats des tests. Enfin, atteindre une couverture de 100 % a requis l'ajout d'un test spécifique pour `main()`, redirigeant `os.Stdout` vers `os.DevNull` afin d'éviter la pollution de la sortie standard pendant l'exécution des tests.
